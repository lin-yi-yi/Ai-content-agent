# v0.4 Agent Architecture

## Positioning

`ai-content-agent` stays a content growth workbench. v0.4 adds a bounded Agent foundation for learning and product practice:

- RAG over imported sources
- LangChain-based document splitting and future tool wrappers
- LangGraph-ready workflow boundary
- explicit data isolation by workspace and knowledge base
- capability allowlist before any tool/action execution
- function-calling style tool execution through an internal allowlist
- local hybrid retrieval with deterministic hash embeddings

It does not become a general desktop agent, browser agent, credential manager, or automatic publisher.

## Product Boundary

In scope:

- read imported `sources` records
- index source content into isolated knowledge bases
- retrieve evidence chunks inside one workspace
- answer with citations or refuse when evidence is weak
- expose current architecture and capability boundaries in the UI
- keep the existing v0.3 content generation workflow working

Out of scope:

- reading arbitrary local files
- using browser cookies or user credentials
- automatic posting to Xiaohongshu, Douyin, GitHub, or other external platforms
- cross-workspace retrieval
- replacing the existing content generation workflow with LangGraph before there is a real branching need

## Data Isolation

New v0.4 tables:

- `workspaces`
- `knowledge_bases`
- `knowledge_documents`
- `knowledge_chunks`

`knowledge_chunks` stores:

- raw chunk text
- token count and chunk index
- local embedding metadata
- `embedding_json` for deterministic 128-dimensional local hash embeddings
- `embedding_provider = local_hash`
- `embedding_model = local-hash-embedding-v1`

Isolation rules:

1. Every RAG document and chunk must include `workspace_id`.
2. Every retrieval request must resolve a `WorkspaceContext`.
3. Every retrieval request must filter by both `workspace_id` and `knowledge_base_id`.
4. The current legacy tables remain unchanged. RAG only reads legacy `sources` by `source_id`.
5. Re-indexing the same `source_id` in the same knowledge base deletes the previous v0.4 document/chunks first.

Current default boundary:

- workspace slug: `default`
- knowledge base name: `内容素材知识库`
- usage: local single-user content and RAG experiments

## Capability Boundary

Capabilities are declared in `backend/app/agent_core/boundaries.py`.

Current allowlist:

- `source_read`: read existing source and chunk data
- `source_index`: write source chunks into a scoped knowledge base
- `rag_retrieve`: retrieve and cite chunks
- `content_generate`: keep existing draft/card/evaluation generation
- `model_call`: call configured model providers with minimal context

Important limitations:

- RAG must refuse when evidence is insufficient.
- Tool execution is not open-ended. New tools should be added to the registry first.
- No capability may read secrets, local files, browser state, or external accounts.
- Generated content remains draft material until human review.

Tool execution is declared in `backend/app/agent_core/tools.py`.

Current tool allowlist:

- `rag.search`: read-only evidence retrieval inside one knowledge base
- `rag.answer`: evidence retrieval plus citation-backed answer/refusal
- `source.index`: limited write path from existing `sources` into scoped RAG tables

Tool execution rules:

1. The API accepts a `tool_name`, optional workspace/knowledge-base ids, and structured `arguments`.
2. The backend rejects unknown tool names before any execution.
3. Each tool maps to one registered capability and action.
4. Each tool validates arguments with a fixed Pydantic schema.
5. Tools cannot receive local file paths, SQL text, Python callables, browser state, credentials, or arbitrary external URLs.

## Framework Layer

LangChain:

- used as an optional adapter in `backend/app/agent_core/langchain_adapter.py`
- current use: `Document` and `RecursiveCharacterTextSplitter`
- fallback: local text splitter if LangChain is not installed

LangGraph:

- detected but not required at runtime
- reserved for future branching workflows such as retrieve -> evaluate evidence -> ask human -> generate
- the current v0.3 `content_growth_agent` remains the production workflow

Retrieval:

- current strategy: local hybrid retrieval
- lexical score: exact term overlap, phrase bonus, density bonus
- vector score: cosine similarity over local hash embeddings
- hybrid score: weighted lexical + vector score
- boundary: still scoped by `workspace_id` and `knowledge_base_id`
- limitation: this is not a production vector database; external embeddings and a vector store remain a next-step upgrade

## API Surface

Architecture:

```text
GET /api/v04/architecture
GET /api/v04/workspaces
GET /api/v04/knowledge-bases
POST /api/v04/knowledge-bases
```

RAG:

```text
POST /api/v04/rag/index-source
POST /api/v04/rag/search
POST /api/v04/rag/answer
```

Tool calling:

```text
GET /api/v04/tools
POST /api/v04/tools/execute
```

Example tool execution:

```json
{
  "tool_name": "rag.search",
  "workspace_id": 1,
  "knowledge_base_id": 1,
  "arguments": {
    "query": "LangChain 和 RAG 在项目里的边界是什么？",
    "top_k": 5,
    "min_score": 0.08
  }
}
```

Agent Run RAG extension:

```text
POST /api/agent-runs
```

Optional request fields:

```json
{
  "use_rag": true,
  "workspace_id": 1,
  "knowledge_base_id": 1,
  "rag_top_k": 5,
  "rag_min_score": 0.08
}
```

When `use_rag` is enabled, the workflow adds a first step:

```text
retrieve_context -> topic_ideas -> create_topic -> score_topic -> generate_draft -> ...
```

The retrieved evidence is stored at:

```text
agent_runs.result_json.rag_context
```

Minimal flow:

1. Import a source through the existing source/topic flow.
2. `POST /api/v04/rag/index-source` with the `source_id`.
3. `POST /api/v04/rag/search` with a query.
4. `POST /api/v04/rag/answer` for a citation-backed answer.

Agent flow with RAG:

1. Index one or more sources from the source library.
2. Enable "知识库检索" in Agent Workbench.
3. Select a knowledge base and start the run.
4. Review the "知识库证据" block before trusting generated content.
5. If evidence is weak, index more source material or treat the output as a viewpoint draft.

## Next Development Order

1. Replace keyword overlap with real embeddings and a vector store.
2. Add LangChain wrappers around the existing allowlisted tools instead of exposing new open-ended executors.
3. Add LangGraph workflow only when branching is needed.
4. Replace FastAPI `BackgroundTasks` with Redis + Celery/RQ/ARQ for durable async tasks.
5. Add tests around workspace isolation and insufficient-evidence refusal.
